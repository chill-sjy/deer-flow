```
Agent 可用工具

│   ├── read_file        读文件内容
```

## 1. 工具加载机制：get_available_tools

**入口**：[`src/tools/tools.py`](../src/tools/tools.py)

```python
def get_available_tools(
    groups: list[str] | None = None,     # 按 group 过滤
    include_mcp: bool = True,            # 是否加载 MCP 工具
    model_name: str | None = None,       # 影响视觉工具是否加入
    subagent_enabled: bool = False,      # 影响 task_tool 是否加入
) -> list[BaseTool]:
```

**加载顺序与过滤逻辑**：

```
config.yaml tools
    ↓ resolve_variable(tool.use, BaseTool)  # 反射：字符串 → 函数对象
    ↓ 按 groups 过滤（web / file:read / file:write / bash）
    ↓
Builtin Tools（固定：present_files, ask_clarification）
    ↓ subagent_enabled=True → 追加 task_tool
    ↓ model.supports_vision=True → 追加 view_image_tool
    ↓
MCP Tools（从 cache 读取，依赖 extensions_config.json 配置）
    ↓
最终列表返回给 Agent
```

`resolve_variable` 是反射机制：把 `"src.community.tavily.tools:web_search_tool"` 这个字符串拆成模块路径和变量名，动态 import 并返回函数对象。类比 Java 的 `Class.forName("...").getMethod(...)`.

**源码**：`src/community/tavily/tools.py`  
**依赖**：Tavily API（需要 API Key）

```python
@tool("web_search", parse_docstring=True)

```

**实现流程**：

```
query → TavilyClient.search(query, max_results=5)
      → 标准化结果（title / url / snippet）
      → JSON 字符串返回给 LLM
```

**config.yaml 可配置**：

```yaml
- name: web_search
  group: web
  use: src.community.tavily.tools:web_search_tool
  max_results: 5          # 返回几条结果
  # api_key: $TAVILY_API_KEY  # 可以在这里写，也可以走环境变量
```

### 2.2 web_fetch —— 网页内容抓取

**源码**：`src/community/jina_ai/tools.py`  
**依赖**：Jina AI Reader（免费，不需要 API Key）

```python
@tool("web_fetch", parse_docstring=True)
def web_fetch_tool(url: str) -> str:
```

**实现流程**：

```
url → JinaClient.crawl(url, return_format="html")
    → ReadabilityExtractor.extract_article(html)   # 提取正文，去广告/导航
    → article.to_markdown()[:4096]                 # 转 Markdown，截断到 4096 字符
```

`ReadabilityExtractor` 是 Mozilla Readability 算法的 Python 实现，专门提取网页正文，过滤掉导航栏、广告、footer 等噪音。

**config.yaml 可配置**：

```yaml
- name: web_fetch
  group: web
  use: src.community.jina_ai.tools:web_fetch_tool
  timeout: 10     # 抓取超时秒数
```

## 3. Sandbox Tools（沙箱执行工具）

**源码**：`src/sandbox/tools.py`

所有 Sandbox Tools 都有一套共同的基础设施，先理解它再看各个工具。

### 3.1 虚拟路径系统（Virtual Paths）

Agent 使用**虚拟路径**操作文件，实际路径在底层替换：

```
虚拟路径（Agent 看到）           实际路径（系统内部）
/mnt/user-data/workspace/*  →  thread_data['workspace_path']/*
/mnt/user-data/uploads/*    →  thread_data['uploads_path']/*
/mnt/user-data/outputs/*    →  thread_data['outputs_path']/*
```

**为什么要虚拟路径？**

- Agent 的路径指令可以跨会话保持一致（永远写 `/mnt/user-data/workspace/`）
- 后端按线程 ID（thread_id）分配独立的实际目录，互不干扰
- 本地模式：`replace_virtual_path()` 在工具调用时做字符串替换
- 容器模式（aio sandbox）：`/mnt/user-data` 直接挂载在容器里，不需要替换

### 3.2 Sandbox 懒加载机制

Sandbox 不在启动时创建，而是**第一次调用 Sandbox Tool 时才获取**（懒加载）：

```python
def ensure_sandbox_initialized(runtime) -> Sandbox:
    # 检查 state 里是否已有 sandbox_id
    sandbox_state = runtime.state.get("sandbox")
    if sandbox_state is not None:
        sandbox = get_sandbox_provider().get(sandbox_id)
        if sandbox is not None:
            return sandbox  # 复用已有 sandbox

    # 没有则懒加载：分配一个新的
    sandbox_id = provider.acquire(thread_id)
    runtime.state["sandbox"] = {"sandbox_id": sandbox_id}  # 存入 state，后续复用
    return sandbox
```

类比 Java：相当于连接池的懒加载，第一次调用时 `pool.acquire()`，之后存在 ThreadLocal 里复用。

### 3.3 bash —— 执行 bash 命令

```python
@tool("bash", parse_docstring=True)
def bash_tool(runtime, description: str, command: str) -> str:
```

**实现流程**：

```
command（可能含虚拟路径）
    → replace_virtual_paths_in_command()   # 批量替换命令里所有 /mnt/user-data/* 路径
    → sandbox.execute_command(command)     # 在沙箱里执行
    → 返回 stdout/stderr 给 LLM
```

**特点**：`description` 参数是必填的，要求 Agent 先解释"为什么要执行这个命令"，提高可解释性和可调试性。

### 3.6 write_file —— 写文件

```python
@tool("write_file", parse_docstring=True)
def write_file_tool(runtime, description, path, content, append=False) -> str:
```

`append=False`（默认）= 覆盖写；`append=True` = 追加写。底层调用 `sandbox.write_file(path, content, append)`，返回 `"OK"` 表示成功。

---

### 3.7 str_replace —— 字符串替换

```python
@tool("str_replace", parse_docstring=True)
def str_replace_tool(runtime, description, path, old_str, new_str, replace_all=False) -> str:
```

**为什么不直接用 write_file 覆盖？**

修改大文件时，LLM 写出完整文件内容会消耗大量 token，还容易出错。`str_replace` 只需传入"要改的那一小段"，更高效、更安全。

```
read_file(path)
    → 检查 old_str 是否唯一存在（replace_all=False 时，不唯一会报错）
    → content.replace(old_str, new_str, 1)
    → write_file(path, new_content)
```

## 4. Builtin Tools（内置工具）

内置工具**不在 config.yaml 里配置**，而是在 `get_available_tools()` 里硬编码加入，具有更深的系统集成能力。

### 4.1 present_files —— 文件呈现

**源码**：`src/tools/builtins/present_file_tool.py`

```python
@tool("present_files", parse_docstring=True)
def present_file_tool(runtime, filepaths: list[str], tool_call_id) -> Command:
```

**特点**：返回值是 LangGraph 的 `Command` 对象，而不是普通字符串。

```python
return Command(
    update={"artifacts": filepaths, ...},
)
```

`Command.update` 直接更新 LangGraph 的状态图（State），把文件路径列表写入 `artifacts` 字段。前端订阅这个字段，渲染成可下载的文件卡片。

**限制**：只能呈现 `/mnt/user-data/outputs/` 下的文件（已生成的输出文件），不能呈现工作目录里的中间文件。

---

### 4.2 ask_clarification —— 向用户请求澄清

**源码**：`src/tools/builtins/clarification_tool.py`

```python
@tool("ask_clarification", parse_docstring=True, return_direct=True)
def ask_clarification_tool(
    question: str,
    clarification_type: Literal["missing_info", "ambiguous_requirement",
                                "approach_choice", "risk_confirmation", "suggestion"],
    context: str | None = None,
    options: list[str] | None = None,
) -> str:
```

**两个关键设计点**：

**① `return_direct=True`**：LangChain 的标记，调用这个工具后，**不把结果再送回 LLM 做下一轮推理**，而是直接把工具结果返回给调用方（最终用户），中断执行流程等待用户回答。

**② 工具本身是"空壳"**：

```python
# 工具函数体只有这一行
return "Clarification request processed by middleware"
```

真正的逻辑在 `ClarificationMiddleware` 里。Middleware 拦截这个工具调用，提取参数，把问题展示给用户，等待用户回答后再恢复执行。

类比 Java：工具本身像一个"钩子方法"，`ClarificationMiddleware` 是 AOP 的切面，真正的处理逻辑在切面里。

**五种澄清类型**：

| clarification_type | 使用场景 |
|---|---|
| `missing_info` | 缺少必要信息（如文件路径） |
| `ambiguous_requirement` | 需求有歧义，有多种合理理解 |
| `approach_choice` | 有多个实现方案，需要用户选 |
| `risk_confirmation` | 高风险操作（删除文件、写 DB）需要确认 |
| `suggestion` | Agent 有建议方案，需要用户批准 |

### 4.3 task —— 派发子任务给 Subagent

**源码**：`src/tools/builtins/task_tool.py`

```python
@tool("task", parse_docstring=True)
def task_tool(runtime, description, prompt, subagent_type, tool_call_id, max_turns=None) -> str:
```

这是系统里最复杂的工具，实现了完整的**异步子任务派发 + 轮询 + 实时流推送**。

**执行流程**：

```
Lead Agent 调用 task_tool(prompt, subagent_type="bash")
    ↓
1. 获取 Subagent 配置（system_prompt / max_turns / timeout）
2. 从 runtime 提取父级上下文（sandbox_state / thread_data / trace_id）
3. 创建 SubagentExecutor，异步启动（executor.execute_async）
    ↓
4. while True 轮询（每 5 秒一次）
    ├── get_background_task_result(task_id)
    ├── 有新 AI 消息 → writer({"type": "task_running", ...})  # 实时推流给前端
    ├── 状态 COMPLETED → writer({"type": "task_completed"})   # 任务完成
    ├── 状态 FAILED    → writer({"type": "task_failed"})      # 任务失败
    └── 状态 TIMED_OUT → writer({"type": "task_timed_out"})   # 超时
    ↓
5. 返回最终结果字符串给 Lead Agent（"Task Succeeded. Result: ..."）
```

**防嵌套设计**：Subagent 调用工具时，`task_tool` 不在可用列表里（`subagent_enabled=False`），防止 Subagent 再派发子任务，避免无限递归。

**`get_stream_writer()`**：LangGraph 提供的流式写入器，可以在工具执行过程中实时向前端推送事件，不用等工具返回。类比 Java 的 `OutputStream.flush()` 在任务进行中就推数据。

**两种 Subagent 类型**：

| subagent_type | 适用场景 |
|---|---|
| `general-purpose` | 复杂多步任务，需要推理+工具组合 |
| `bash` | 命令执行专家，适合 git / 构建 / shell 脚本 |

### 4.4 view_image —— 读取图片

**源码**：`src/tools/builtins/view_image_tool.py`

```python
@tool("view_image", parse_docstring=True)
def view_image_tool(runtime, image_path: str, tool_call_id) -> Command:
```

**仅对视觉模型开放**（`model_config.supports_vision=True` 时才加入工具列表）。

**实现流程**：

```
image_path（虚拟路径）
    → replace_virtual_path()        # 替换为实际路径
    → 校验（路径合法 / 文件存在 / 格式支持）
    → open(path, "rb") → base64.b64encode()
    → Command(update={"viewed_images": {path: {"base64": ..., "mime_type": ...}}})
```

图片以 **base64 编码**写入 State 的 `viewed_images` 字段，LangGraph 在下次 LLM 调用时，把这些图片以 multimodal 消息的形式注入，让视觉模型"看到"图片内容。

## 5. 设计模式总结

| 模式 | 体现 |
|---|---|
| **虚拟路径抽象** | Agent 使用 `/mnt/user-data/*` 统一路径，后端按 thread_id 映射实际路径 |
| **Sandbox 懒加载** | 第一次调用 Sandbox Tool 时才 `acquire()`，避免启动时占用资源 |
| **空壳工具 + Middleware** | `ask_clarification` 工具本身是空的，逻辑在 Middleware 里；解耦工具定义与执行 |
| **Command 更新 State** | `present_files`、`view_image` 返回 `Command` 直接修改 LangGraph State，而不是普通字符串 |
| **流式推送** | `task_tool` 轮询中用 `get_stream_writer()` 实时推送进度，用户不需要等任务结束才看到反馈 |
| **防递归 Subagent** | Subagent 的工具列表里没有 `task_tool`，从根本上防止无限嵌套 |
