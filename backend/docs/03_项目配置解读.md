## 1. summarization_config.py —— 对话历史自动总结

### 背景

大模型有上下文窗口限制（如 128k tokens）。对话越来越长时，历史消息会超出限制导致报错或被截断。`SummarizationConfig` 控制**何时自动压缩历史、压缩后保留多少**，让对话可以无限持续。

### ContextSize：计量单位

总结的触发阈值和保留量，都用 `ContextSize` 描述，支持三种计量方式：

```python
ContextSizeType = Literal["fraction", "tokens", "messages"]

class ContextSize(BaseModel):
    type: ContextSizeType   # 计量方式
    value: int | float      # 数值
```

| type | 含义 | 示例 |
|---|---|---|
| `"messages"` | 按消息条数 | `value=50` → 50条消息 |
| `"tokens"` | 按 token 数量 | `value=4000` → 4000 tokens |
| `"fraction"` | 占模型上限的比例 | `value=0.8` → 占用80%上限 |

### SummarizationConfig 字段详解

```python
class SummarizationConfig(BaseModel):
    enabled: bool = False
    # 默认关闭。对话长度可控时不需要开启，长对话场景才开启。

    model_name: str | None = None
    # 做总结用哪个模型。总结不需要很强的模型，None = 自动选轻量模型，节省成本。

    trigger: ContextSize | list[ContextSize] | None = None
    # 触发总结的阈值，满足任意一个就触发。
    # 支持单个或多个阈值，多个时任一满足即触发（OR 关系）。

    keep: ContextSize = ContextSize(type="messages", value=20)
    # 总结后保留多少最近历史。不能全删，要留一些近期上下文给模型参考。

    trim_tokens_to_summarize: int | None = 4000
    # 给模型做总结时，最多喂多少 tokens 的历史内容。
    # 太长会超出总结模型的上限，所以做一次裁剪再总结。

    summary_prompt: str | None = None
    # 自定义总结 prompt 模板。None = 使用 LangChain 内置默认 prompt。
```

### 工作流程

```
对话进行中...
  消息数: 1, 5, 10 ... 50 ← 达到 trigger 阈值，触发总结！
        ↓
  取最近 4000 tokens 的历史（trim_tokens_to_summarize 裁剪）
        ↓
  用 summary_prompt 让模型生成一段摘要文字
        ↓
  清理旧历史，保留最近 20 条消息（keep）+ 摘要注入 system prompt
        ↓
  继续对话，上下文被压缩，不超出模型限制
```

### 典型配置示例

```python
# 消息超过50条，或达到模型容量80%时触发总结，总结后保留最近20条
config = SummarizationConfig(
    enabled=True,
    trigger=[
        ContextSize(type="messages", value=50),
        ContextSize(type="fraction", value=0.8),
    ],
    keep=ContextSize(type="messages", value=20),
)
```

## 2. memory_config.py —— 用户记忆机制

### 背景：Memory 存的不是聊天记录，是"用户档案"

> **常见误解**：以为 Memory 是把聊天消息原文存起来。  
> **实际上**：Memory 是通过 LLM **提取结构化事实**，建立关于用户本人的持久化档案。

防抖触发后，系统把对话喂给 LLM，LLM 的任务是：

```
对话内容（原始消息）
      ↓  LLM 事实提取
{
  "user": {
    "workContext": "Java 开发者，正在学习 Python...",
    "personalContext": "偏好分步骤、详细讲解...",
    "topOfMind": "当前在学习异步编程..."
  },
  "newFacts": [
    { "content": "用户是 Java 开发者", "confidence": 0.95 },
    { "content": "用户偏好 Java 技术类比", "confidence": 0.90 }
  ]
}
```

LLM **不是在写摘要**，而是在回答"这个用户是谁、有什么偏好、近期在关注什么"——输出是结构化 JSON，落盘到 `memory.json`，下次对话仍可用。

### Memory vs Summarization：本质区别

两者都用了 LLM，但做的是**完全不同的事**：

| | SummarizationConfig | MemoryConfig |
|---|---|---|
| **LLM 在做什么** | 压缩当前对话历史（"我们聊了什么"） | 提取用户档案事实（"这个用户是谁"） |
| **输出形式** | 一段自然语言摘要文字 | 结构化 JSON（user/history/facts） |
| **作用范围** | 仅限当前这次对话 | 跨对话，永久保存 |
| **触发时机** | 消息数/token 数超过阈值（对话进行中） | 对话安静 30 秒后（防抖） |
| **生命周期** | 当前对话结束即清除 | 永久保存，下次对话仍可用 |
| **注入方式** | 替换掉旧消息，摘要作为上下文保留 | 注入 system prompt 的 `<memory>` 块 |

**一句话总结**：
- Summarization = "帮我把这段对话总结一下，省点 token"
- Memory = "从这段对话里，了解这个用户是什么样的人"

### MemoryConfig 字段详解

```python
class MemoryConfig(BaseModel):
    enabled: bool = True
    # 是否开启记忆机制，默认开启。

    storage_path: str = ""
    # 记忆数据存储路径（JSON 文件）。
    # 空字符串 = 默认路径 {base_dir}/memory.json
    # 支持绝对路径和相对路径（相对于 base_dir，不是 backend 工作目录）

    debounce_seconds: int = 30   # 范围：1~300
    # 防抖时间（秒）。对话安静 30 秒后，才触发一次 LLM 事实提取。
    # 避免每条消息都触发一次昂贵的 LLM 调用。

    model_name: str | None = None
    # 更新记忆用哪个模型。None = 使用默认模型。

    max_facts: int = 100   # 范围：10~500
    # 最多存储多少条事实。超出后按置信度排序，淘汰低置信度的事实，防止无限膨胀。

    fact_confidence_threshold: float = 0.7   # 范围：0.0~1.0
    # 事实置信度阈值。模型判断某个信息的置信度低于 0.7 时，不存入记忆。
    # 类比：不把"用户可能喜欢咖啡"这种不确定信息存进去，只存确定的事实。

    injection_enabled: bool = True
    # 是否把记忆注入 system prompt。
    # True = 每次对话开始时，把相关记忆拼到 system prompt 里，让模型"记得"用户。

    max_injection_tokens: int = 2000   # 范围：100~8000
    # 注入记忆时最多占用多少 tokens。
    # 避免记忆太多把模型的有效上下文全占了，影响当前对话质量。
```

### 关键字段深入：debounce_seconds（防抖）

`debounce` 是防抖，来自前端概念：

```
消息到来:  A --- B --- C --- D --- (安静 30 秒) → 触发 LLM 事实提取！
            ↓     ↓     ↓     ↓
          重置   重置   重置   重置定时器
                                         ↑
                              只有真正停止聊天后，才执行一次
```

**防抖触发后，LLM 做的事**（对照 `updater.py`）：

```python
# 1. 把对话格式化成文本
conversation_text = format_conversation_for_update(messages)

# 2. 喂给 LLM，提示词是 MEMORY_UPDATE_PROMPT
#    任务：分析对话，更新用户档案，提取事实
prompt = MEMORY_UPDATE_PROMPT.format(
    current_memory=json.dumps(current_memory),   # 已有的用户档案
    conversation=conversation_text,              # 新的对话内容
)

# 3. LLM 返回结构化 JSON（不是摘要文字！）
#    包含：用户背景更新、新事实列表、需要删除的旧事实
update_data = json.loads(response.content)

# 4. 合并到 memory.json，永久保存
_save_memory_to_file(updated_memory)
```

类比 Java：`ScheduledExecutorService` 做延迟任务，每次有新消息就 `cancel()` 旧任务、重新 `schedule()` 一个 30 秒后的任务。只有真正"安静了 30 秒"，才触发 LLM 事实提取并写入 JSON 文件。

### 关键字段深入：fact_confidence_threshold（存储时过滤）

LLM 提取事实时，会给每条事实打置信度分数，低于阈值的直接丢弃：

```
对话："我好像喜欢喝咖啡，不确定"  → 置信度 0.4 → 低于 0.7 → 不存入
对话："我是一名 Java 开发者"      → 置信度 0.95 → 高于 0.7 → 存入记忆
```

### 关键字段深入：max_facts（存储层截断）

存储时有上限，超出后**按置信度排序，低分事实先被淘汰**（对照 `updater.py`）：

```python
if len(current_memory["facts"]) > config.max_facts:
    current_memory["facts"] = sorted(
        current_memory["facts"],
        key=lambda f: f.get("confidence", 0),
        reverse=True,
    )[:config.max_facts]   # 只保留置信度最高的 top N 条
```

不是"先进先出"，而是"弱者先出"——置信度低的事实优先被淘汰。

### 关键字段深入：injection_enabled + max_injection_tokens（注入层截断）

**存储** 和 **注入** 是两个独立阶段，`max_injection_tokens` 只影响注入，不影响存储：

```
[JSON 文件]  max_facts=100 条         [System Prompt]  max_injection_tokens=2000 tokens
    ↓ 每次对话开始时注入                    ↑ 超出则硬截断
    全量记忆（可能 5000+ tokens）  →  截取前 2000 tokens + "\n..."
```

截断逻辑（对照 `prompt.py`）：

```python
token_count = _count_tokens(result)   # tiktoken 精确计数
if token_count > max_tokens:
    char_per_token = len(result) / token_count
    target_chars = int(max_tokens * char_per_token * 0.95)  # 留 5% margin
    result = result[:target_chars] + "\n..."   # 硬截断 + 省略号
```

注入到 system prompt 的效果：

```
System Prompt:
  你是一个智能助手。
  <memory>
  User Context:
  - Work: Java 开发者，正在学习 Python，deer-flow 项目贡献者
  - Personal: 偏好分步骤、详细讲解，喜欢 Java 技术类比
  - Current Focus: 学习 Python 异步机制和项目配置解读
  </memory>
```

| 阶段 | 配置项 | 截断策略 | 影响范围 |
|---|---|---|---|
| 写入 JSON | `max_facts` | 按置信度排序，低分淘汰 | 永久影响存储 |
| 注入 prompt | `max_injection_tokens` | 字符硬截断 + `\n...` | 仅影响当次注入，JSON 不变 |

### 典型配置示例

```python
# 严格模式：只存高置信度事实，记忆注入限制在 1000 tokens
config = MemoryConfig(
    enabled=True,
    max_facts=50,
    fact_confidence_threshold=0.85,
    max_injection_tokens=1000,
    debounce_seconds=60,
)
```

## 3. subagents_config.py —— 子 Agent 超时控制

### 背景：什么是 Subagent？

主 Agent（Lead Agent）在处理复杂任务时，会把子任务派发给若干 **Subagent** 并行执行（类似线程池里的工作线程）。每个 Subagent 独立运行、耗时不定，因此需要超时机制防止某个 Subagent 卡死拖垮整体。

`SubagentsAppConfig` 提供两层超时控制：**全局默认超时** + **按 Agent 名称的独立覆盖**。

### SubagentsAppConfig 字段详解

```python
class SubagentOverrideConfig(BaseModel):
    """单个 Agent 的超时覆盖配置。"""
    timeout_seconds: int | None = None
    # None = 不覆盖，使用全局默认值
    # 设置具体数字 = 只有这个 Agent 用自己的超时


class SubagentsAppConfig(BaseModel):
    """整个子 Agent 系统的配置。"""

    timeout_seconds: int = 900   # 默认 900 秒 = 15 分钟
    # 全局默认超时。所有 Subagent 的 fallback 值。
    # 15 分钟是个保守值，足够大多数任务完成，同时防止无限等待。

    agents: dict[str, SubagentOverrideConfig] = {}
    # 按 Agent 名称的独立超时配置。
    # key = agent 名称（字符串），value = SubagentOverrideConfig
    # 示例：{"bash_agent": SubagentOverrideConfig(timeout_seconds=60)}
```

### 超时查找逻辑：get_timeout_for

核心方法 `get_timeout_for` 实现了**"有覆盖用覆盖，没覆盖用全局"**的两级查找：

```python
def get_timeout_for(self, agent_name: str) -> int:
    override = self.agents.get(agent_name)   # 先查有没有专属配置
    if override is not None and override.timeout_seconds is not None:
        return override.timeout_seconds      # 有 → 用专属超时
    return self.timeout_seconds              # 没有 → 用全局默认
```

**两级优先级**：

```
config.yaml 配置：
  subagents:
    timeout_seconds: 900          ← 全局默认（level 2）
    agents:
      bash_agent:
        timeout_seconds: 60       ← bash_agent 专属覆盖（level 1，优先）
      heavy_analysis_agent:
        timeout_seconds: 1800     ← 某个重型 Agent 给更长时间

查询结果：
  get_timeout_for("bash_agent")           → 60    （用专属覆盖）
  get_timeout_for("heavy_analysis_agent") → 1800  （用专属覆盖）
  get_timeout_for("general_agent")        → 900   （没覆盖，用全局默认）
```

类比 Java：相当于一个 `Map<String, Integer>` 存各 Agent 的专属超时，`getOrDefault(agentName, globalTimeout)` 查找。

### 配置单例与加载机制

```python
# 模块级单例，进程内全局唯一
_subagents_config: SubagentsAppConfig = SubagentsAppConfig()  # 默认值初始化

def get_subagents_app_config() -> SubagentsAppConfig:
    return _subagents_config   # 直接返回单例

def load_subagents_config_from_dict(config_dict: dict) -> None:
    global _subagents_config
    _subagents_config = SubagentsAppConfig(**config_dict)  # 从 config.yaml 覆盖
```

这是项目里所有配置文件共用的模式：
1. 模块加载时用默认值初始化单例
2. 应用启动时从 `config.yaml` 读取并调用 `load_xxx_from_dict()` 覆盖
3. 业务代码调用 `get_xxx_config()` 获取全局单例，不需要传参

### 典型配置示例（config.yaml 写法）

```yaml
subagents:
  timeout_seconds: 900          # 全局默认 15 分钟
  agents:
    bash_agent:
      timeout_seconds: 60       # bash 命令一般很快，60 秒足够，快速失败
    file_analysis_agent:
      timeout_seconds: 1800     # 大文件分析可能很慢，给 30 分钟
```
