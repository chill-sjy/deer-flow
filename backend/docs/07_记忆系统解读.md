## 0. 先回答：这个项目有几种"记忆"？

很多人第一次看到这个项目会以为只有一种记忆。其实是**两套完全不同的系统**，解决的是两个不同的问题：

```
记忆系统
├── 长期记忆（Long-term Memory）    ← 跨会话的用户档案，持久化到磁盘
│   └── src/agents/memory/
│
└── 上下文压缩（Summarization）     ← 当前会话内的 context window 管理
    └── langchain.agents.middleware.SummarizationMiddleware
```

| 维度 | 长期记忆 | 上下文压缩 |
|---|---|---|
| **目标** | 记住"用户是谁" | 防止 context window 溢出 |
| **存储位置** | `memory.json`（磁盘文件） | State["messages"]（内存） |
| **跨会话** | ✅ 是 | ❌ 否（当前会话重置） |
| **触发时机** | `after_agent`（每轮对话结束后） | `before_model`（每次 LLM 调用前） |
| **数据形态** | 结构化 JSON（事实 + 摘要） | 替换掉的旧消息 → 一条 HumanMessage |
| **LLM 作用** | 事实提取 + 用户画像构建 | 对话内容压缩摘要 |

**类比 Java**：
- 长期记忆 ≈ 数据库里的用户档案表（`UserProfile`），进程重启后依然存在
- 上下文压缩 ≈ 内存里的 LRU 缓存（`LRUCache`），进程退出就丢失

## 1. 长期记忆系统

### 1.1 它解决什么问题？

没有长期记忆的 AI 助手，每次对话都是"失忆状态"——你告诉它你是 Java 开发者，下次对话它完全不知道。

长期记忆的核心目标：**让 Agent 在每次新对话开始时，知道"这个用户是谁"**。

```
第 1 次对话（周一）：
    用户："我是一名 Java 开发者，正在学 Python"
    → 提取事实 → memory.json → { "content": "用户是 Java 开发者", "confidence": 0.95 }

第 2 次对话（周三）：
    用户："帮我理解这段 Python 代码"
    → 系统提示中注入记忆 →
    → Agent 知道用户是 Java 背景 → 自动用 Java 术语类比 Python 概念
```

### 1.2 数据结构：memory.json

长期记忆存储在一个结构化 JSON 文件里，默认路径为 `{base_dir}/memory.json`：

```json
{
  "version": "1.0",
  "lastUpdated": "2026-03-12T10:30:00Z",
  "user": {
    "workContext": {
      "summary": "Java 开发者，正在学习 Python 和 LangGraph 相关框架",
      "updatedAt": "2026-03-10T08:00:00Z"
    },
    "personalContext": {
      "summary": "偏好详细的步骤讲解，喜欢用 Java 术语类比",
      "updatedAt": "2026-03-11T09:00:00Z"
    },
    "topOfMind": {
      "summary": "当前在学习 LangGraph Agent 架构，关注异步编程和中间件模式，同时调研 DeerFlow 项目的记忆机制",
      "updatedAt": "2026-03-12T10:00:00Z"
    }
  },
  "history": {
    "recentMonths": {
      "summary": "最近 1-3 个月：学习 Python 异步编程，深入研究 LangChain v1 的 create_agent API...",
      "updatedAt": "2026-03-12T10:30:00Z"
    },
    "earlierContext": {
      "summary": "3-12 个月前：主要从事 Java Spring Boot 后端开发，开始探索 AI 相关技术...",
      "updatedAt": "2026-03-01T00:00:00Z"
    },
    "longTermBackground": {
      "summary": "长期背景：Java 后端工程师，有丰富的 Spring 生态经验，2024 年开始学习 AI 开发",
      "updatedAt": "2026-02-01T00:00:00Z"
    }
  },
  "facts": [
    {
      "id": "fact_a1b2c3d4",
      "content": "用户是 Java 开发者",
      "category": "context",
      "confidence": 0.95,
      "createdAt": "2026-03-10T08:00:00Z",
      "source": "thread_abc123"
    },
    {
      "id": "fact_e5f6g7h8",
      "content": "用户偏好通过 Java 术语类比来理解 Python 概念",
      "category": "preference",
      "confidence": 0.90,
      "createdAt": "2026-03-11T09:00:00Z",
      "source": "thread_def456"
    }
  ]
}
```

**字段含义**：

| 字段 | 类型 | 含义 | 更新频率 |
|---|---|---|---|
| `user.workContext` | 摘要 | 当前工作背景（职位、项目、技术栈），2-3 句话 | 变动时更新 |
| `user.personalContext` | 摘要 | 语言、沟通偏好、兴趣，1-2 句话 | 变动时更新 |
| `user.topOfMind` | 摘要 | **当前多个并发的关注点**，3-5 句话（最高频更新） | 每轮对话更新 |
| `history.recentMonths` | 段落 | 近 1-3 个月活动，4-6 句话 | 按时间轴更新 |
| `history.earlierContext` | 段落 | 3-12 个月前的规律，3-5 句话 | 偶尔更新 |
| `history.longTermBackground` | 段落 | 不变的基础背景，2-4 句话 | 很少更新 |
| `facts` | 列表 | **离散的具体事实**，带置信度 | 新增/删除 |

`facts` 中每条的 `category` 字段含义：

| category | 含义 | 示例 |
|---|---|---|
| `preference` | 用户的偏好/厌恶 | "偏好分步骤讲解" |
| `knowledge` | 用户掌握的技术/领域 | "熟悉 Spring Boot" |
| `context` | 背景事实（职位/项目/地点） | "用户是 Java 开发者" |
| `behavior` | 行为模式/工作习惯 | "倾向于先看整体架构" |
| `goal` | 目标/学习计划 | "正在学习 AI 开发" |

### 1.3 整体流程

```
对话结束（after_agent 触发）
    ↓
MemoryMiddleware.after_agent()
    ↓ 过滤消息（只保留 HumanMessage + 无 tool_calls 的 AIMessage）
    ↓ 丢弃 ToolMessage、带 tool_calls 的 AIMessage（中间推理步骤）
    ↓
get_memory_queue().add(thread_id, filtered_messages)
    ↓
MemoryUpdateQueue（防抖队列）
    ↓ 如果同一 thread_id 已有待处理任务 → 替换（去重）
    ↓ 重置 Timer（30 秒后执行，每次新增都重新计时）
    ↓
30 秒内无新消息 → Timer 触发 _process_queue()
    ↓ 后台线程执行（不阻塞主流程）
    ↓
MemoryUpdater.update_memory(messages, thread_id)
    ↓ 读取当前 memory.json（带文件 mtime 缓存）
    ↓ format_conversation_for_update() 格式化对话
    ↓ 构建 MEMORY_UPDATE_PROMPT（携带现有记忆 + 新对话）
    ↓ 调用 LLM → 返回结构化 JSON
    ↓ _apply_updates() 合并更新
    ↓ _strip_upload_mentions_from_memory() 清理文件上传记录
    ↓ 原子写入（先写 .tmp → rename 到 .json）
    ↓ 更新内存缓存（mtime 同步）

下次对话开始（apply_prompt_template 调用）
    ↓
_get_memory_context()
    ↓ get_memory_data()（读缓存，mtime 变化则重新加载）
    ↓ format_memory_for_injection()（格式化 + tiktoken 精确 token 计数）
    ↓ 如果超过 max_injection_tokens → 按 token 比例截断 + "\n..."
    ↓ 包装成 <memory>...</memory> 标签
    ↓ 注入到 SYSTEM_PROMPT_TEMPLATE 的 {memory_context} 槽位
```

### 1.4 防抖队列：为什么要防抖？

**问题背景**：调用 LLM 做事实提取是有成本的（时间 + 钱）。如果用户快速连续发了 5 条消息，每条消息结束后都立刻触发 LLM 更新，就会产生 5 次不必要的重复更新。

**解决方案**：防抖（Debounce）— 不是"每次触发立刻执行"，而是"最后一次触发后等待 N 秒，没有新触发才执行"。

```
用户第 1 条消息结束 → 加入队列 → 开始计时 30s
用户第 2 条消息结束（10s 后）→ 队列去重（替换同 thread_id 的旧任务）→ 重置计时 30s
用户第 3 条消息结束（15s 后）→ 队列去重 → 重置计时 30s
... 用户停止对话 ...
30s 后 → 执行一次 LLM 更新（只用最新的消息状态）
```

**类比 Java**：就像 Swing 里的 `Timer.restart()`——每次用户输入都重置计时器，只在用户停止输入后才执行搜索。或者前端的输入框防抖（`debounce(fn, 300)`）。

**核心代码**：

```python
# src/agents/memory/queue.py

def add(self, thread_id: str, messages: list[Any], ...) -> None:
    with self._lock:
        # 同一 thread_id 有待处理任务 → 替换（保证只处理最新消息）
        self._queue = [c for c in self._queue if c.thread_id != thread_id]
        self._queue.append(context)

        # 重置定时器（每次添加都重新计时）
        self._reset_timer()

def _reset_timer(self) -> None:
    # 取消旧定时器
    if self._timer is not None:
        self._timer.cancel()

    # 启动新定时器（daemon=True → 进程退出时自动销毁）
    self._timer = threading.Timer(
        config.debounce_seconds,  # 默认 30s
        self._process_queue,
    )
    self._timer.daemon = True
    self._timer.start()
```

**并发安全**：使用 `threading.Lock` 保护所有对队列的读写，同时用 `self._processing` 标志防止重入（如果上一批还没处理完，新触发会重新排队）。

### 1.5 LLM 事实提取：Prompt 设计

`MEMORY_UPDATE_PROMPT` 是长期记忆的核心。它把两样东西喂给 LLM：

1. **现有的 memory.json 内容**（让 LLM 知道"已经知道了什么"）
2. **新的对话片段**（让 LLM 分析"新学到了什么"）

LLM 的输出是一个**严格的 JSON 结构**，声明哪些字段需要更新：

```json
{
  "user": {
    "workContext": { "summary": "...", "shouldUpdate": true },
    "personalContext": { "summary": "...", "shouldUpdate": false },
    "topOfMind": { "summary": "...", "shouldUpdate": true }
  },
  "history": {
    "recentMonths": { "summary": "...", "shouldUpdate": true },
    "earlierContext": { "summary": "...", "shouldUpdate": false },
    "longTermBackground": { "summary": "...", "shouldUpdate": false }
  },
  "newFacts": [
    { "content": "用户喜欢防抖设计模式", "category": "preference", "confidence": 0.85 }
  ],
  "factsToRemove": ["fact_a1b2c3d4"]
}
```

**`shouldUpdate` 的意义**：LLM 在每个字段上都声明是否真的有新信息。如果某次对话只聊了技术问题，`personalContext` 很可能是 `shouldUpdate: false`，系统就不会覆盖它，避免无意义的更新。

**置信度过滤**（`fact_confidence_threshold`，默认 0.7）：

```
对话："我觉得我可能是 Java 开发者？" → LLM 给 0.4 分 → 低于 0.7 → 丢弃（不确定性太高）
对话："我在公司用 Java 写了 5 年代码" → LLM 给 0.97 分 → 高于 0.7 → 存入
```

**max_facts 淘汰策略**：当事实数量超过上限（默认 100 条）时，不是先进先出，而是**置信度优先淘汰**：

```python
# 按置信度排序，只保留 top N 条
current_memory["facts"] = sorted(
    current_memory["facts"],
    key=lambda f: f.get("confidence", 0),
    reverse=True,
)[: config.max_facts]
```

### 1.6 消息过滤：什么内容会被记忆？

`MemoryMiddleware` 并不把完整的 `State["messages"]` 喂给 LLM，而是先过滤：

```
完整消息历史（State["messages"]）
├── HumanMessage("帮我理解这段代码")      ✅ 保留
├── AIMessage(tool_calls=[...])            ❌ 丢弃（中间推理步骤）
├── ToolMessage("执行结果: ...")            ❌ 丢弃（工具调用结果）
├── ToolMessage("执行结果: ...")            ❌ 丢弃
├── AIMessage("根据结果，这段代码的作用是...") ✅ 保留（最终回答）
└── HumanMessage("<uploaded_files>...</uploaded_files>只看这个文件") ⚠️ 特殊处理
    → 剥离 <uploaded_files> 标签，只保留 "只看这个文件"
    → 如果剥离后内容为空 → 完全丢弃此消息及配对的 AI 回复
```

**为什么要丢弃文件上传记录？**

文件上传是**会话级的**——文件上传到 `/mnt/user-data/uploads/xxx.pdf`，下次新会话这个路径就不存在了。如果把 "用户上传了 xxx.pdf" 记入长期记忆，下次对话时 Agent 可能会尝试读取一个不存在的文件，造成混乱。

项目用了**两道防线**：
1. `_filter_messages_for_memory()`：过滤阶段剥离 `<uploaded_files>` 标签
2. `_strip_upload_mentions_from_memory()`：保存前再扫描一遍，用正则把所有提及上传的句子从摘要里删掉

### 1.7 原子写入与缓存失效

文件写入用了**原子操作**，防止并发写入导致文件损坏：

```python
# 先写临时文件
temp_path = file_path.with_suffix(".tmp")
with open(temp_path, "w") as f:
    json.dump(memory_data, f, indent=2)

# rename 是原子操作（大多数文件系统保证）
# 即使程序在这一步崩溃，要么是旧文件，要么是新文件，不会出现半写状态
temp_path.replace(file_path)
```

**缓存机制**（`_memory_cache`）：读取 `memory.json` 很频繁（每次对话都要注入），但文件 I/O 有开销。项目用了基于文件 mtime 的缓存：

```python
# 检查文件修改时间
current_mtime = file_path.stat().st_mtime

# 如果 mtime 没变 → 直接返回缓存
if cached[1] == current_mtime:
    return cached[0]

# mtime 变了（文件被更新）→ 重新加载
memory_data = _load_memory_from_file()
_memory_cache[agent_name] = (memory_data, current_mtime)
```

这保证了：LLM 后台更新了 `memory.json` 后，下次对话自动读到最新内容，无需显式通知。

### 1.8 per-agent 记忆 vs 全局记忆

系统支持两种粒度的记忆存储，取决于 `agent_name` 参数：

```python
# 全局记忆（所有对话共享）
MemoryMiddleware(agent_name=None)
→ 存储路径：{base_dir}/memory.json

# per-agent 记忆（某个 Agent 专属）
MemoryMiddleware(agent_name="coding-assistant")
→ 存储路径：{base_dir}/agents/coding-assistant/memory.json
```

Lead Agent 使用全局记忆，让所有对话共享同一份用户档案。不同的自定义 Agent 可以有各自的记忆文件，互不干扰。

## 2. 上下文压缩（Summarization）

### 2.1 它解决什么问题？

大模型有上下文窗口限制（如 128k tokens）。当一次对话持续很长时，消息历史会超出限制：

```
第 1 条消息（100 tokens）
第 2 条消息（200 tokens）
...
第 80 条消息 → 总计超过 128k tokens → LLM 报错或丢失早期上下文
```

**上下文压缩**的解决方案：当历史消息接近上限时，把较早的消息压缩成一段摘要，只保留最近的原始消息：

```
压缩前：[消息1, 消息2, ..., 消息60, 消息61, ..., 消息80]（太长）
压缩后：[摘要(消息1-60), 消息61, ..., 消息80]（合理长度）
```

### 2.2 与长期记忆的本质区别

```
长期记忆：跨会话的用户档案
  → 关心的是：这个用户是谁？有什么偏好？
  → 存储：磁盘文件，进程重启后依然有效
  → 粒度：用户维度

上下文压缩：当前会话的 context window 管理
  → 关心的是：这次对话的早期内容是什么？
  → 存储：State["messages"]（内存），会话结束即丢弃
  → 粒度：会话维度
```

### 2.3 触发机制

`SummarizationMiddleware` 挂在 `before_model` 钩子——**每次调用 LLM 之前**都会检查一次是否需要压缩。支持三种触发条件（任意一个满足即触发，OR 逻辑）：

```yaml
trigger:
  - type: tokens      # 消息总 token 数超过 4000
    value: 4000
  - type: messages    # 消息总条数超过 50 条
    value: 50
  - type: fraction    # 消息总 token 数占模型上限的 80%
    value: 0.8
```

| trigger 类型 | 含义 | 适用场景 |
|---|---|---|
| `tokens` | 超过固定 token 数 | 大多数场景，最直观 |
| `messages` | 超过固定消息条数 | 消息很短的场景 |
| `fraction` | 超过模型上限的百分比 | 多模型混用时自适应 |

### 2.4 压缩流程详解

```
before_model 触发检查
    ↓
token 计数（approximate，Anthropic 模型：~3.3 字符/token，其他：LangChain 默认估算）
    ↓
是否超过 trigger 阈值？
    ↓ 是
消息分区（根据 keep 策略）
    ├── 需要压缩的部分：较早的消息（超出 keep 范围的）
    └── 需要保留的部分：最近的 N 条消息 / N tokens / N%（keep 配置决定）

    ⚠️ AI/Tool Pair 保护机制：
    如果切割点落在某个 AI 消息的 ToolMessage 内部，
    系统会自动往前移动切割点，确保一组 AI+ToolMessage 不被拆开

    ↓
调用 LLM 对"需要压缩的部分"生成摘要
    ↓ 输入给 LLM 的消息先用 trim_tokens_to_summarize 截断（默认 4000 tokens）
    ↓ 防止摘要调用本身消耗太多 token

    ↓
替换消息历史：
旧：[msg1, msg2, ..., msg60, msg61, ..., msg80]
新：[HumanMessage("Here is a summary: [摘要文本]"), msg61, ..., msg80]

    ↓ Checkpointer 自动持久化更新后的 State
```

**摘要消息的格式**（注入为 HumanMessage）：

```
Here is a summary of the conversation to date:

用户询问了如何配置 LangGraph agent，讨论了 create_agent 的五个参数（model、tools、
middleware、system_prompt、state_schema）。随后深入探讨了 Middleware 钩子机制，
特别是 before_model 和 after_model 的使用场景...
```

### 2.5 配置参考

```yaml

  enabled: false                   # 默认关闭，需要手动开启
  model_name: null                 # 压缩用的 LLM（null = 用默认模型，建议用轻量模型节省成本）
  trigger:                         # 触发条件（以下任一满足即触发）
    - type: tokens
      value: 4000
    - type: messages
      value: 50
  keep:                            # 保留策略（压缩后保留多少原始消息）
    type: messages
    value: 20
  trim_tokens_to_summarize: 4000  # 喂给摘要 LLM 的最大 token 数
  summary_prompt: null             # 自定义摘要 Prompt（null = 用 LangChain 默认）
```

**keep 策略说明**：

```yaml
# 按消息条数保留
keep:
  type: messages
  value: 20        # 保留最近 20 条消息

# 按 token 数保留
keep:
  type: tokens
  value: 3000      # 保留最近 3000 tokens 的消息

# 按模型上限比例保留
keep:
  type: fraction
  value: 0.3       # 保留占模型上限 30% 的消息
```

## 3. 两套系统如何协作

以下是一次完整的"长对话"全生命周期，展示两套记忆系统如何互补：

```
────────── 会话 1（今天）──────────

用户：你好，我是 Java 开发者，想学 LangGraph
    ↓
before_model → SummarizationMiddleware 检查：消息少，不触发
    ↓
LLM 回答...
    ↓
after_agent → MemoryMiddleware 触发：把对话加入防抖队列

（... 对话持续，消息越来越多 ...）

第 50 条消息到来：
    ↓
before_model → SummarizationMiddleware 触发！（超过 50 条消息）
    ↓ 把第 1-30 条消息压缩成一段摘要
    ↓ State["messages"] 变为：[摘要消息, 消息31, ..., 消息50]
    ↓ LLM 只看到压缩后的消息历史，不超限
    ↓
对话继续...

会话 1 结束，30 秒后：
    ↓
MemoryUpdateQueue 触发后台更新
    ↓ LLM 从过滤后的对话中提取事实：
    ↓ → "用户是 Java 开发者" (confidence: 0.95)
    ↓ → "正在学 LangGraph" (confidence: 0.90)
    ↓ → "偏好 Java 术语类比" (confidence: 0.88)
    ↓ 写入 memory.json

────────── 会话 2（明天）──────────

新对话开始：
    ↓
apply_prompt_template()
    ↓ _get_memory_context() 读取 memory.json
    ↓ 注入到系统提示：
    │ <memory>
    │ User Context:
    │ - Work: Java 开发者，正在学习 LangGraph...
    │ - Current Focus: 研究 LangGraph 中间件机制...
    │ </memory>
    ↓
LLM 知道"这个用户是 Java 背景"
    ↓ 自动用 Java 类比讲解概念，无需用户重复介绍自己
```

## 4. 设计决策剖析

### 4.1 为什么用 JSON 文件而不是数据库？

| 方案 | 优点 | 缺点 |
|---|---|---|
| JSON 文件（当前方案） | 零依赖、部署简单、人类可读 | 并发写入需要额外处理 |
| SQLite | 并发安全、支持查询 | 多一个依赖，部署稍复杂 |
| 向量数据库 | 支持语义检索 | 重量级依赖，小场景杀鸡用牛刀 |

项目的定位是**易部署的开源工具**，JSON 文件方案最符合目标用户的使用场景。原子写入（temp + rename）解决了并发问题。

### 4.2 为什么摘要消息格式是 HumanMessage 而不是 SystemMessage？

`SummarizationMiddleware` 把摘要注入为 `HumanMessage`（以 "Here is a summary..." 开头），而不是 `SystemMessage`。

原因：大多数 LLM 对消息历史里出现 `SystemMessage` 的处理比较敏感（部分模型不允许历史中有多条 SystemMessage），而 `HumanMessage` 是最通用、兼容性最好的格式。

### 4.3 长期记忆为什么不用向量检索？

当前实现是全量注入（把整个 memory.json 的主要内容注入 prompt）而不是向量检索相关片段。

原因：用户档案本身不大（几百到 2000 tokens），向量检索增加了系统复杂度，但对于这个量级的数据没有明显收益。`max_injection_tokens` 参数控制注入量上限，超出则截断。

## 5. 设计模式总结

| 模式 | 体现位置 | 说明 |
|---|---|---|
| **防抖模式** | `MemoryUpdateQueue` | 收集最后一次操作，忽略短时间内的中间状态，减少 LLM 调用次数 |
| **单例模式** | `get_memory_queue()` | 全局一个队列实例，所有 MemoryMiddleware 共享同一防抖器 |
| **AOP（切面编程）** | `MemoryMiddleware.after_agent` | 把记忆更新与业务逻辑解耦，不侵入对话主流程 |
| **缓存 + mtime 失效** | `_memory_cache` | 避免频繁文件读取，同时保证文件更新后自动刷新 |
| **原子写入** | `temp_path.replace(file_path)` | 写文件不会出现中间状态，防止崩溃导致文件损坏 |
| **策略模式** | `ContextSizeType` | 触发和保留策略支持 tokens/messages/fraction 三种，可配置切换 |
| **过滤器模式** | `_filter_messages_for_memory` | 只让"对记忆有意义"的消息通过，过滤掉噪声（tool call 中间步骤） |
