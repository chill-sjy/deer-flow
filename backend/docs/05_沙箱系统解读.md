# 沙箱系统解读

> 覆盖 `src/sandbox/`、`src/community/aio_sandbox/`、`src/config/sandbox_config.py`、`src/config/paths.py`

---

## 0. 先回答：Sandbox 跟 Docker 有什么关系？

Sandbox（沙箱）是这个项目的一个**抽象概念**，它有两种具体实现：

| 实现 | 有没有 Docker | 原理 |
|---|---|---|
| **LocalSandbox**（本地沙箱） | **没有** Docker | 直接用 `subprocess.run()` 在宿主机上执行命令 |
| **AioSandbox**（容器沙箱） | **有** Docker（或 Apple Container） | 启动一个 Docker 容器，通过 HTTP API 在容器内执行命令 |

你在开发机上跑的极大概率是 **LocalSandbox**（config.yaml 默认配置）。AioSandbox 是生产/部署用的隔离方案。

---

## 1. 为什么需要沙箱？问题到底出在哪？

Agent 执行任务时，会调用 `bash_tool` 来运行真实的 shell 命令，比如：

```bash
python data_analysis.py
pip install pandas
rm -rf /tmp/old_files
```

这些命令**直接在你的操作系统上跑**。问题来了：

**问题1：多个对话线程并发执行，文件互相污染**

想象两个用户同时在用这个系统：
- 线程A 正在写 `/tmp/output.csv`
- 线程B 也在写 `/tmp/output.csv`
- 结果：两个线程的文件内容互相覆盖，数据乱掉

**问题2：Agent 写的文件去哪了？用户找不到**

Agent 生成了一份报告，存到了 `/tmp/report.pdf`，但 `/tmp` 是系统临时目录，重启就没了，用户根本不知道文件在哪。

**问题3：危险命令无法隔离（生产环境尤其严重）**

Agent 误判执行了 `rm -rf /home/someuser`，真实的用户文件就没了。

---

## 2. 沙箱怎么解决这些问题的？不是复制文件！

一个非常关键的问题：**沙箱不是把系统目录复制一份给每个线程用**，而是给每个对话线程分配一个**专属目录**，然后用**路径映射**让 Agent 感知不到差异。

**每个线程在磁盘上有自己的专属目录：**

```
~/.deer-flow/threads/
├── thread-abc123/
│   └── user-data/
│       ├── workspace/   # Agent 写代码、处理数据的工作区
│       ├── uploads/     # 用户上传的文件
│       └── outputs/     # Agent 产出的结果文件
└── thread-xyz789/
    └── user-data/
        ├── workspace/
        ├── uploads/
        └── outputs/
```

**Agent 看到的是统一的虚拟路径：**

```
/mnt/user-data/workspace/
/mnt/user-data/uploads/
/mnt/user-data/outputs/
```

Agent 永远只写 `/mnt/user-data/...`，不同线程的这个路径在**背后对应不同的真实目录**，所以天然隔离，互不干扰。

这就是"沙箱"的本质——**不是隔离文件，而是隔离访问路径**。

## 3. 整体架构：三层抽象

```
┌─────────────────────────────────────────────────────────┐
│  Tools 层（bash_tool / ls_tool / read_file_tool...）    │  ← Agent 调用工具
│  只管调 ensure_sandbox_initialized(runtime)             │
└────────────────────────┬────────────────────────────────┘
                         │ 拿到 Sandbox 对象，调 execute_command / read_file / ...
┌────────────────────────▼────────────────────────────────┐
│  SandboxProvider 层（接口）                              │  ← 管理沙箱生命周期
│  acquire(thread_id) → sandbox_id                        │
│  get(sandbox_id) → Sandbox                              │
│  release(sandbox_id)                                     │
│                                                          │
│  实现 A: LocalSandboxProvider（单例，开发用）            │
│  实现 B: AioSandboxProvider（Docker 容器，生产用）       │
└────────────────────────┬────────────────────────────────┘
                         │ 返回具体 Sandbox 实例
┌────────────────────────▼────────────────────────────────┐
│  Sandbox 层（接口）                                      │  ← 实际执行能力
│  execute_command(command) → str                          │
│  read_file(path) → str                                   │
│  write_file(path, content, append)                       │
│  list_dir(path, max_depth) → list[str]                   │
│  update_file(path, content: bytes)                       │
│                                                          │
│  实现 A: LocalSandbox → 调 subprocess.run()             │
│  实现 B: AioSandbox → 调 HTTP API（到容器内）            │
└─────────────────────────────────────────────────────────┘
```

这是典型的**策略模式（Strategy Pattern）**。`SandboxProvider` 和 `Sandbox` 都是抽象接口（`ABC`），运行时根据 config.yaml 的 `sandbox.use` 字段，通过**反射**决定用哪个实现类。

**反射入口**：[`src/sandbox/sandbox_provider.py`](../src/sandbox/sandbox_provider.py)

```python
def get_sandbox_provider(**kwargs) -> SandboxProvider:
    config = get_app_config()
    cls = resolve_class(config.sandbox.use, SandboxProvider)  # 字符串 → 类对象
    _default_sandbox_provider = cls(**kwargs)
    return _default_sandbox_provider
```

## 4. 实现 A：LocalSandbox（本地沙箱）

**源码**：[`src/sandbox/local/local_sandbox.py`](../src/sandbox/local/local_sandbox.py)

### 4.1 LocalSandbox 是什么？

`LocalSandbox` 的本质是：**一个带路径翻译能力的命令执行器**。

执行一条命令的完整流程：

```
Agent 调用 bash_tool("python /mnt/user-data/workspace/script.py")
    ↓
LocalSandbox._resolve_paths_in_command() 做路径替换：
"python /Users/you/.deer-flow/threads/abc123/user-data/workspace/script.py"
    ↓
subprocess.run() 用系统 shell 执行真实命令
    ↓
输出里如果含真实路径，再反向替换回虚拟路径返回给 Agent
```

`subprocess.run()` 就是 Python 调用操作系统 shell 执行命令的标准方式，等价于你在终端手敲那条命令。这里没有任何魔法，就是**路径翻译 + 执行命令**。

### 4.2 核心代码

```python
def execute_command(self, command: str) -> str:
    # 第一步：把命令里的虚拟路径替换成宿主机真实路径
    resolved_command = self._resolve_paths_in_command(command)

    # 第二步：直接用系统 shell 执行
    result = subprocess.run(
        resolved_command,
        executable=self._get_shell(),   # /bin/zsh → /bin/bash → /bin/sh 按顺序找
        shell=True,
        capture_output=True,
        text=True,
        timeout=600,                    # 最多等 10 分钟
    )

    # 第三步：拼接 stdout + stderr，输出里的真实路径再反向替换回虚拟路径
    output = result.stdout
    if result.stderr:
        output += f"\nStd Error:\n{result.stderr}"
    return self._reverse_resolve_paths_in_output(output)
```

### 4.3 bash_tool：Agent 调用命令的入口

[`bash_tool`](../src/sandbox/tools.py) 是暴露给 Agent LLM 的工具函数，它是"门面"，`LocalSandbox` 是"执行者"：

```python
@tool("bash")
def bash_tool(runtime, description: str, command: str) -> str:
    sandbox = ensure_sandbox_initialized(runtime)   # 1. 拿到当前线程对应的沙箱
    ensure_thread_directories_exist(runtime)         # 2. 确保专属目录已创建
    if is_local_sandbox(runtime):
        command = replace_virtual_paths_in_command(command, thread_data)  # 3. 路径翻译
    return sandbox.execute_command(command)          # 4. subprocess 执行
```

两者分工：`bash_tool` 负责从 runtime 上下文里取状态、做路径替换；`LocalSandbox.execute_command()` 只管拿到命令执行并返回结果。

### 4.4 LocalSandboxProvider：谁来管沙箱的生命周期？

每个线程需要一个沙箱，谁来负责"分配"和"查找"？这是 [`LocalSandboxProvider`](../src/sandbox/local/local_sandbox_provider.py) 的职责。

```python
_singleton: LocalSandbox | None = None

class LocalSandboxProvider(SandboxProvider):
    def acquire(self, thread_id: str | None = None) -> str:
        global _singleton
        if _singleton is None:
            _singleton = LocalSandbox("local", path_mappings=self._path_mappings)
        return _singleton.id   # 永远返回 "local"

    def get(self, sandbox_id: str) -> Sandbox | None:
        return _singleton
```

**为什么所有线程共用同一个 LocalSandbox 实例（单例）？**

因为 LocalSandbox 的线程隔离靠的是**路径翻译规则**——每个线程的虚拟路径翻译到不同的真实目录（`threads/abc123/` vs `threads/xyz789/`）。这个翻译逻辑在 `bash_tool` 里按线程各自处理，不需要每个线程创建一个独立的 `LocalSandbox` 实例。单例的 `LocalSandbox` 实例可以服务所有线程。

## 5. 实现 B：AioSandbox（容器沙箱）

**源码**：[`src/community/aio_sandbox/aio_sandbox.py`](../src/community/aio_sandbox/aio_sandbox.py)

### 5.1 核心原理：HTTP API 转发

```python
class AioSandbox(Sandbox):
    def __init__(self, id: str, base_url: str, ...):
        self._client = AioSandboxClient(base_url=base_url, timeout=600)
        # base_url 例如 http://localhost:8081

    def execute_command(self, command: str) -> str:
        result = self._client.shell.exec_command(command=command)
        return result.data.output
```

`AioSandboxClient` 是 `agent_sandbox` 库提供的 HTTP 客户端。命令通过 HTTP 发到容器内运行的 API 服务器（端口 8080），在**容器内**执行，结果通过 HTTP 响应返回。

### 5.2 AioSandboxProvider：Docker 容器生命周期管理

**源码**：[`src/community/aio_sandbox/aio_sandbox_provider.py`](../src/community/aio_sandbox/aio_sandbox_provider.py)

这是整个沙箱系统里最复杂的部分，管理 Docker 容器的完整生命周期。

#### 三层一致性机制（acquire 的核心逻辑）

```
acquire(thread_id="aaa")
    ↓
Layer 1: 进程内 dict 缓存（最快）
    _thread_sandboxes["aaa"] → "sandbox-abc123"
    如果存在且 Sandbox 实例还在 → 直接返回
    ↓（缓存未命中）
Layer 2: 跨进程文件锁 + 状态文件
    FileSandboxStateStore.load("aaa")
    如果文件里有记录 → backend.discover() 验证容器是否还活着
    如果容器活着 → 复用，写回内存缓存
    ↓（文件里也没有 / 容器已死）
Layer 3: 创建新容器
    backend.create(thread_id, sandbox_id, extra_mounts)
    等待容器 ready（HTTP 健康检查，最多 60 秒）
    把 sandbox_id 写入文件，写入内存缓存
    ↓
返回 sandbox_id
```

**为什么要三层？** 多进程部署时（如 Gunicorn 多 worker），不同 worker 可能服务同一个 thread_id 的不同请求。三层机制保证同一 thread_id 始终用同一个容器。

类比 Java：类似 `ConcurrentHashMap` + 数据库持久化 + 服务发现的组合，保证分布式一致性。

#### 容器启动命令示例

`LocalContainerBackend._start_container()` 实际执行的是：

```bash
docker run \
  --security-opt seccomp=unconfined \
  --rm -d \
  -p 8081:8080 \
  --name deer-flow-sandbox-abc123 \
  -v /Users/you/.deer-flow/threads/aaa/user-data/workspace:/mnt/user-data/workspace \
  -v /Users/you/.deer-flow/threads/aaa/user-data/uploads:/mnt/user-data/uploads \
  -v /Users/you/.deer-flow/threads/aaa/user-data/outputs:/mnt/user-data/outputs \
  -v /Users/you/.deer-flow/skills:/mnt/skills:ro \
  enterprise-public-cn-beijing.cr.volces.com/vefaas-public/all-in-one-sandbox:latest
```

关键点：
- `--rm`：容器停止后自动删除（不留垃圾）
- `-d`：后台运行
- `-p 8081:8080`：把容器内的 8080 端口映射到宿主机随机空闲端口
- `-v`：把宿主机的线程专属目录挂载进容器（宿主机和容器共享文件，没有复制）

#### macOS 特殊支持：Apple Container

```python
def _detect_runtime(self) -> str:
    if platform.system() == "Darwin":
        try:
            subprocess.run(["container", "--version"], ...)
            return "container"   # Apple Container（更轻量，原生 macOS）
        except:
            pass
    return "docker"
```

macOS 上优先用 **Apple Container**（苹果自家容器运行时），没有的话 fallback 到 Docker。

#### 空闲超时回收

```python
DEFAULT_IDLE_TIMEOUT = 600  # 10 分钟

# 后台 daemon 线程，每 60 秒检查一次
def _idle_checker_loop(self):
    while not stop_event.wait(timeout=60):
        for sandbox_id, last_activity in self._last_activity.items():
            if time.time() - last_activity > idle_timeout:
                self.release(sandbox_id)  # 停掉 Docker 容器，释放端口
```

类比 Java 数据库连接池：超过最大空闲时间就自动关闭连接，释放资源。

## 6. 虚拟路径系统（Virtual Paths）

这是让 LocalSandbox 和 AioSandbox 在路径上行为一致的关键机制。

### 6.1 路径映射规则

**源码**：[`src/config/paths.py`](../src/config/paths.py) + [`src/sandbox/tools.py`](../src/sandbox/tools.py)

```
Agent 使用的虚拟路径            →    宿主机实际路径
/mnt/user-data/workspace/*  →  ~/.deer-flow/threads/{thread_id}/user-data/workspace/*
/mnt/user-data/uploads/*    →  ~/.deer-flow/threads/{thread_id}/user-data/uploads/*
/mnt/user-data/outputs/*    →  ~/.deer-flow/threads/{thread_id}/user-data/outputs/*
```

### 6.2 两种模式处理路径的差异

**LocalSandbox 模式**（需要手动替换）：

```python
# tools.py 里，每个工具调用前：
if is_local_sandbox(runtime):
    command = replace_virtual_paths_in_command(command, thread_data)
    # /mnt/user-data/workspace/a.py → /Users/you/.deer-flow/threads/aaa/.../a.py
```

**AioSandbox 模式**（不需要替换）：

Docker 启动时已经通过 `-v` 参数把宿主机目录挂载到容器的 `/mnt/user-data/`，因此容器内直接访问 `/mnt/user-data/workspace/` 就等同于访问宿主机对应目录，**路径本身就是真实的**。

| 模式 | 路径替换 | 原因 |
|---|---|---|
| LocalSandbox | 需要（字符串替换） | 宿主机没有 `/mnt/user-data`，必须换成真实路径 |
| AioSandbox | 不需要 | 容器里 `/mnt/user-data` 直接挂载，路径即真实 |

### 6.3 为什么要统一用虚拟路径？

Agent 的 Prompt 里写的路径永远是 `/mnt/user-data/workspace/`，**与具体部署方式无关**。无论你换 Local 还是 Docker 模式，Agent 不用改一行指令。这就是**关注点分离**。

## 7. 懒加载机制（Lazy Initialization）

沙箱不是在 Agent 启动时就创建，而是**第一次调用 Sandbox Tool 时才创建**。

**源码**：[`src/sandbox/tools.py`](../src/sandbox/tools.py) `ensure_sandbox_initialized()`

```python
def ensure_sandbox_initialized(runtime) -> Sandbox:
    # 检查 state 里是否已有 sandbox_id
    sandbox_state = runtime.state.get("sandbox")
    if sandbox_state is not None:
        sandbox_id = sandbox_state.get("sandbox_id")
        if sandbox_id:
            sandbox = get_sandbox_provider().get(sandbox_id)
            if sandbox is not None:
                return sandbox  # 复用已有 sandbox

    # 第一次调用：懒加载
    thread_id = runtime.context.get("thread_id")
    provider = get_sandbox_provider()
    sandbox_id = provider.acquire(thread_id)           # ← 这里才真正创建/获取

    runtime.state["sandbox"] = {"sandbox_id": sandbox_id}  # 存入 State，后续复用
    return provider.get(sandbox_id)
```

**流程**：

```
第 1 次调用 bash_tool
    ↓ ensure_sandbox_initialized()
    ↓ state["sandbox"] 为空 → 触发懒加载
    ↓ provider.acquire(thread_id) → 分配沙箱
    ↓ sandbox_id 写入 state["sandbox"]
    ↓ 执行命令

第 2、3、N 次调用 bash_tool（同一个 thread）
    ↓ ensure_sandbox_initialized()
    ↓ state["sandbox"]["sandbox_id"] = "local"（或某个哈希 ID）
    ↓ 直接 provider.get(sandbox_id) → 拿到已有沙箱
    ↓ 执行命令（无额外开销）
```

类比 Java：就像 Spring 的 `@Lazy` Bean，第一次被注入时才实例化，之后复用。

## 8. SandboxMiddleware：Agent 钩子

**源码**：[`src/sandbox/middleware.py`](../src/sandbox/middleware.py)

```python
class SandboxMiddleware(AgentMiddleware):
    def before_agent(self, state, runtime) -> dict | None:
        if self._lazy_init:
            return super().before_agent(state, runtime)  # 跳过，交给工具懒加载

        # lazy_init=False 时：在 Agent 开始前就立刻 acquire
        if "sandbox" not in state or state["sandbox"] is None:
            thread_id = runtime.context["thread_id"]
            sandbox_id = provider.acquire(thread_id)
            return {"sandbox": {"sandbox_id": sandbox_id}}
```

`SandboxMiddleware` 挂在 Agent 执行链上，负责在合适时机初始化 sandbox state。默认 `lazy_init=True`，沙箱创建推迟到第一次工具调用。

## 9. 整体调用链

```
用户发消息
    ↓
SandboxMiddleware.before_agent()
    ↓  lazy_init=True 时跳过，等到工具调用再说
    ↓
Agent LLM 决定执行命令，调用 bash_tool
    ↓
bash_tool
    ↓  ensure_sandbox_initialized() → 从 state 找 sandbox_id → Provider.get() → Sandbox 实例
    ↓  ensure_thread_directories_exist() → 确保专属目录已创建
    ↓  replace_virtual_paths_in_command() → 虚拟路径翻译成真实路径
    ↓
LocalSandbox.execute_command()
    ↓  subprocess.run() 执行真实命令
    ↓  反向翻译输出里的真实路径 → 虚拟路径
    ↓
返回结果给 Agent
```

## 10. 异常体系

**源码**：[`src/sandbox/exceptions.py`](../src/sandbox/exceptions.py)

```
SandboxError（基类）
├── SandboxNotFoundError        沙箱 ID 找不到（可能已被回收）
├── SandboxRuntimeError         运行时配置错误（没有 thread_id 等）
├── SandboxCommandError         命令执行失败（exit_code 非 0）
├── SandboxFileError            文件操作失败
│   ├── SandboxPermissionError  权限不足
│   └── SandboxFileNotFoundError 文件不存在
```

所有 Sandbox Tool（bash_tool 等）都有统一的 `except SandboxError as e: return f"Error: {e}"` 兜底，把异常转换成 LLM 可读的字符串错误信息，不会导致 Agent 崩溃。

## 12. 目录结构：线程数据如何存储

宿主机上实际的目录结构（LocalSandbox 模式）：

```
~/.deer-flow/
├── memory.json
├── USER.md
└── threads/
    └── {thread_id}/
        └── user-data/        ← Docker 模式会把这整个目录挂进容器
            ├── workspace/    ← /mnt/user-data/workspace/
            ├── uploads/      ← /mnt/user-data/uploads/
            └── outputs/      ← /mnt/user-data/outputs/
```

`Paths` 类（[`src/config/paths.py`](../src/config/paths.py)）提供了所有路径计算的单一入口，避免路径字符串散落在代码各处。

## 14. 设计模式总结

| 模式 | 体现位置 | 说明 |
|---|---|---|
| **策略模式** | `Sandbox(ABC)`、`SandboxProvider(ABC)` | 工具层只依赖接口，运行时通过反射注入实现 |
| **单例模式** | `LocalSandboxProvider._singleton`、`get_sandbox_provider()` | Provider 和 LocalSandbox 都是全局单例 |
| **懒初始化** | `ensure_sandbox_initialized()` | 首次工具调用时才 acquire 沙箱，避免启动时浪费资源 |
| **路径虚拟化** | `replace_virtual_path()`、`-v` Docker 挂载 | Agent 写死虚拟路径，底层适配具体部署；两种实现方式不同但对 Agent 透明 |
| **三层一致性** | `AioSandboxProvider._acquire_internal()` | 内存缓存 → 文件锁状态 → 容器发现，保证多进程共享同一容器 |
| **Middleware 钩子** | `SandboxMiddleware.before_agent()` | 在 Agent 执行链切面里做沙箱初始化，解耦生命周期管理与业务逻辑 |
| **空闲超时回收** | `AioSandboxProvider._idle_checker_loop()` | 后台 daemon 线程定期扫描，回收长期空闲的 Docker 容器 |
